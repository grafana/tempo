# Tempo 3.0 changelog — sorted by topic

Source: [PR #7103](https://github.com/grafana/tempo/pull/7103) (v3.0.0-rc.0 CHANGELOG section)

## Proposed top features (confirm with engineering team)

1. **New architecture is production-ready** — Ingesters removed. Live store + block builder is now the only ingest path. The `all` target is 3.0-compatible. Single-binary mode pushes directly to live-store without Kafka. Project Rhythm graduates from experimental to default.
2. **TraceQL metrics GA** — TraceQL metrics move to general availability. Adds comparison operators in metrics queries, a faster read path (experimental), step interval alignment with vParquet5, and increases default max query duration to one day. Enables use cases like alerting on trace-derived metrics.
3. **Migration tooling for 2.x to 3.0** — `tempo-cli migrate config` command, `migrate overrides-config` / `migrate overrides-per-tenant` commands, legacy overrides disabled by default. Comprehensive breaking changes require migration planning.
4. **Span profiling** — New otelpyroscope integration attaches pprof labels to OTel spans. Enables span-level profiling.
5. **Metrics-generator cardinality management** — Per-label limiter, include_any filter policy, span multiplier from W3C tracestate. Continues improved cardinality control from 2.10.
6. **vParquet5 as default block format** — vParquet5 becomes the default block format (was optional in 2.10). Brings blob detection for high-cardinality attributes, dedicated columns for arrays and events, LLM-optimized API responses, and doubled dedicated string column limits. The format is optimized for both human and AI-driven trace analysis.

## vParquet5 as default block format — summary

vParquet5 becomes the default block format in Tempo 3.0. It was introduced as optional in 2.10 and is now production-ready and the default for all new blocks.

### What's new in 3.0

These changes are new in the 3.0 changelog:

* Doubled the maximum number of dedicated string columns and updated tempo-cli to determine the optimum number for the data. [#6282](https://github.com/grafana/tempo/pull/6282)
* Removed explicit `runtime.GC()` calls in vParquet5 compactor/block creation and CLI, reducing GC overhead. [#6603](https://github.com/grafana/tempo/pull/6603)
* TraceQL metrics step intervals now align with vParquet5 timestamp columns for better query performance. [#6413](https://github.com/grafana/tempo/pull/6413)
* Bug fixes for dedicated column filtering, buffer-reuse in event attributes, and dedicated column fallback in block builder/live store. [#6586](https://github.com/grafana/tempo/pull/6586), [#6914](https://github.com/grafana/tempo/pull/6914), [#6647](https://github.com/grafana/tempo/pull/6647)

### Capabilities from 2.10 now enabled by default

These vParquet5 capabilities shipped in 2.10 as opt-in and are now the default experience:

* **Dedicated columns for array values** — Efficient storage for multi-valued attributes like HTTP headers. [#5760](https://github.com/grafana/tempo/pull/5760)
* **Blob detection** — Automatically identifies high-cardinality attributes (UUIDs, stack traces) and applies optimized encoding to reduce memory pressure and improve performance. [#5946](https://github.com/grafana/tempo/pull/5946)
* **Event-level dedicated columns** — Dedicated columns now support event-level attributes, not just span and resource attributes. [#5946](https://github.com/grafana/tempo/pull/5946)
* **Virtual span row numbers** — Speeds up queries that filter only on resource attributes. [#5943](https://github.com/grafana/tempo/pull/5943)
* **Flexible dedicated column defaults** — Replace the previous well-known attribute columns with configurable defaults. [#5696](https://github.com/grafana/tempo/pull/5696)
* **Native `span:childCount` storage** — TraceQL's `span:childCount` intrinsic is stored natively in vParquet5 for faster lookups. [#6126](https://github.com/grafana/tempo/pull/6126)
* **Automatic cleanup** — Unused dedicated columns are automatically removed to save storage. [#5946](https://github.com/grafana/tempo/pull/5946)

### AI and LLM integration

vParquet5's data layout improvements also benefit AI-driven trace analysis:

* **LLM-optimized API responses** — The `Accept: application/vnd.grafana.llm` header returns a simplified JSON structure that flattens complex OpenTelemetry format into a more readable representation with computed values like duration in milliseconds. Used automatically by the Tempo MCP server. [#5962](https://github.com/grafana/tempo/pull/5962)
* **Blob detection for AI context** — Optimized encoding for high-cardinality string attributes (stack traces, UUIDs) means AI assistants can query traces with large attribute values without excessive memory or performance cost.

### Upgrade note

vParquet5 is now the default. No migration of existing data is required — Tempo reads existing vParquet4 blocks while writing new blocks in vParquet5 format. The `v2` block encoding is removed entirely in 3.0. vParquet3 and vParquet2 were deprecated in 2.10 and are also removed.

## AI and LLM integration — summary

Tempo 3.0 is the first release where the full AI/LLM stack works out of the box. Features introduced across 2.9, 2.10, and 3.0 come together to make Tempo a first-class data source for AI-driven observability.

### The AI feature chain across releases

#### Introduced in 2.9

- **MCP server (experimental)** — Gives AI assistants like Claude Code and Cursor direct access to tracing data. Exposes 7 tools (`traceql-search`, `traceql-metrics-instant`, `traceql-metrics-range`, `get-trace`, `get-attribute-names`, `get-attribute-values`, `docs-traceql`) and 4 TraceQL documentation resources. [PR 5212](https://github.com/grafana/tempo/pull/5212), [documentation](/docs/tempo/<TEMPO_VERSION>/api_docs/mcp-server/)

#### Introduced in 2.10

- **LLM-optimized API responses** — `Accept: application/vnd.grafana.llm` header returns simplified JSON that flattens the complex OTel format into a readable representation with computed values like duration in milliseconds. The MCP server uses this format automatically. [PR 5962](https://github.com/grafana/tempo/pull/5962)
- **MCP server CLI flag** (2.10.4) — `--query-frontend.mcp-server.enabled=true` makes it easier to enable in Docker and demo environments without YAML changes. [PR 6903](https://github.com/grafana/tempo/pull/6903)
- **vParquet5 blob detection** — Optimized encoding for high-cardinality string attributes (stack traces, UUIDs) so AI queries over large attribute values don't cause excessive memory or performance cost. [PR 5946](https://github.com/grafana/tempo/pull/5946)

#### New or elevated in 3.0

- **vParquet5 as default** — All LLM-friendly storage optimizations (blob detection, array columns, flexible dedicated columns) are now on by default. No configuration needed.
- **TraceQL metrics GA** — The MCP server's `traceql-metrics-instant` and `traceql-metrics-range` tools are now backed by a production-ready metrics engine. Enables AI-driven alerting, capacity analysis, and performance monitoring workflows.
- **OR conditions for tag autocomplete** — AI assistants using the MCP server's `get-attribute-names` / `get-attribute-values` tools benefit from more flexible autocomplete queries. [#6827](https://github.com/grafana/tempo/pull/6827)
- **tempo-cli `--header` flag** — Enables passing custom headers (like auth tokens) when using the CLI programmatically, useful for scripted AI workflows. [#6768](https://github.com/grafana/tempo/pull/6768)
- **tempo-cli `redact` command** — Submit trace redaction jobs to the backend scheduler, supporting data governance for AI/LLM use cases where sensitive data must be redacted before exposure. [#6832](https://github.com/grafana/tempo/pull/6832)
- **TraceRedactor `ErrTraceHidden`** — Hide entire traces from query results, supporting data governance policies around what trace data AI assistants can access. [#6811](https://github.com/grafana/tempo/pull/6811)
- **Span profiling** — Attaches pprof labels to OTel spans via otelpyroscope, giving AI assistants richer context when analyzing traces by connecting trace spans to profiling data. [#7063](https://github.com/grafana/tempo/pull/7063)

### Why 3.0 for AI/LLM adoption

| Layer | What it provides | Release |
|---|---|---|
| **Interface** | MCP server — standardized protocol for AI assistants to query traces | 2.9 |
| **Format** | LLM-optimized API — simplified JSON responses AI can parse accurately | 2.10 |
| **Storage** | vParquet5 default — blob detection, array columns, optimized encoding | 3.0 |
| **Metrics** | TraceQL metrics GA — production-ready metrics tools in MCP server | 3.0 |
| **Governance** | TraceRedactor + redact command — control what data AI can access | 3.0 |
| **Context** | Span profiling — richer trace data with profiling links | 3.0 |

In 3.0, these layers work together without extra configuration. Enable the MCP server, and AI assistants get production-ready access to traces, metrics, and attribute discovery with optimized response formats and data governance controls.
